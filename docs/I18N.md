# FearlessMath — Multi-language (i18n) plan & pipeline

FearlessMath supports a **global language toggle** (English / हिंदी / తెలుగు). Translated
packs reuse the **same concept ids** as English, so a child's progress, unlock graph and
answer-checking are shared across languages, and any untranslated lesson **falls back to
English per-concept** automatically.

Only **prose** is translated. Numbers, answers, formulas, KaTeX, `mistakeTag`s, concept
ids, prerequisites and every numeric visual prop stay **byte-identical** to English — so a
translation can change how a lesson reads, never what it teaches or how it is graded.

---

## Rollout order — class by class, youngest first

Translate a **whole class band** before moving up, rather than scattering concepts. A
child learns within a band; finishing one gives a gap-free mother-tongue path (no jarring
English tile mid-journey). This also matches NEP 2020, which pushes mother-tongue hardest
in the Foundational/Preparatory stages.

| Batch | Band | ~Concepts | Priority |
|------|------|-----------|----------|
| 1 | Foundation (PP1–2) + Class 3–5 | ~55 | **Highest** — youngest, largest audience |
| 2 | Class 6–8 | ~40 | High |
| 3 | Class 9–10 (board) | ~40 | Medium |
| 4 | Class 11–12 + Olympiad | ~46 | Low — these learners read English comfortably |

Each band ships independently thanks to English fallback; you never need all 181 done at once.

Proving slice already shipped (`hi` + `te`): `oly-modular-arithmetic`, `oly-pigeonhole`,
`oly-invariants-parity`.

## Pipeline — how a band is produced

The pipeline **guarantees the math survives**: the script owns the JSON structure and only
sends prose strings to the LLM, then writes translations back into a clone of the English
concept.

```
# 1. Dry run (no API key) — see how many strings a band needs
node tools/translate-pack.js --src content-packs/cbse-pp1-2-en-v1/concepts --lang hi --dry

# 2. Translate a band (OpenAI-compatible provider via env)
set FM_TRANSLATE_KEY=sk-...              # PowerShell: $env:FM_TRANSLATE_KEY="sk-..."
node tools/translate-pack.js ^
  --src content-packs/cbse-pp1-2-en-v1/concepts ^
  --lang hi --out content-packs/cbse-pp1-2-hi-v1/concepts

# 3. Add the band's pack.json (language: "hi" / "te"), then prove parity
npm run validate-i18n

# 4. Human review — WORDING ONLY (structure + answers already guaranteed)

# 5. Build + ship
npm run dist
```

Provider env vars: `FM_TRANSLATE_KEY` (required), `FM_TRANSLATE_URL`
(default `https://api.openai.com/v1/chat/completions`), `FM_TRANSLATE_MODEL`
(default `gpt-4o-mini`). Any OpenAI-compatible endpoint works (incl. an Indic-tuned model).

Useful flags: `--only id1,id2` (subset), `--limit N` (smoke test), `--overwrite`
(replace existing), `--dry` (extract + count, no LLM, no writes).

---

## The two guardrails

1. **Structure-preserving pipeline** — `tools/translate-pack.js` copies ids, grades,
   prerequisites, numeric answers, `mistakeTag`s, visual components and numeric props
   verbatim; the model only ever returns translated prose strings.
2. **Parity validator** — `tools/validate-i18n.js` (`npm run validate-i18n`) fails the
   build if any translated concept diverges from English on ids, question structure,
   answers or mistakeTag keys. Machine output cannot corrupt the math.

Because of these two, **human review is wording-only** and fast.

---

## Terminology glossary

`tools/i18n-glossary/<lang>.json` locks English→target math terms (remainder → शेषफल / శేషం,
etc.). The pipeline injects the glossary into every request so a band stays consistent.
**Extend** the glossary as new bands introduce terms; **never change** an existing mapping
once a band has shipped.

---

## Packaging

Mirror the English pack layout per language: `cbse-pp1-2-hi-v1`, `cbse-pp1-2-te-v1`,
`cbse-class3-5-hi-v1`, `cbse-class3-5-te-v1`, … Each folder needs a `pack.json` with the
right `"language"`. The loader groups packs by language and merges within a language, so
the folder split is only for tidiness.

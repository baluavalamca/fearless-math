The Algorithms Every Student Must Know

A tour of the algorithms that show up in every data-structures course, every coding interview, and most real software. For each one: the core idea, how it works step by step, the exact time & space cost, clean code, and when to reach for it — plus live visualizers you can play with.

00 What is an algorithm?

An algorithm is a finite, unambiguous, step-by-step procedure that transforms an input into a correct output. Think of it as a recipe a computer can follow with zero guessing.

Three questions to ask about any algorithm

  • Is it correct? Does it produce the right answer for every valid input, including empty, single-element, and duplicate cases?
  • How fast is it? How does the running time grow as the input n grows? This is time complexity.
  • How much memory? Extra space used beyond the input is space complexity.

Great engineers don't memorise code — they memorise ideas and trade-offs. The same handful of patterns (divide & conquer, greedy choices, remembering sub-answers, exploring a graph) reappear everywhere. Learn the patterns and you can rebuild the code on demand.

01 Big-O & complexity — the language of speed

Big-O describes how an algorithm's cost grows as input size n grows, ignoring constants and small terms. It answers "will this still work when n is a million?"

02 Linear search

Linear (sequential) search O(n)O(1) space

Idea: walk through the list one item at a time until you find the target (or run out). The only option when data is unsorted.

Best O(1)Average O(n)Worst O(n)Space O(1)
python
def linear_search(arr, target):
    for i, x in enumerate(arr):
        if x == target:
            return i          # found — return the index
    return -1                 # not found
Use when data is unsorted, small, or you only search once.
Avoid when data is sorted or searched repeatedly — use binary search or a hash set.

03 Binary search ★ interview-critical

On a sorted array, repeatedly look at the middle element and throw away half the search space. This is the single most important "halving" idea in all of computing.

🔎 Live: step through binary search

The array is sorted. Pick a target, then press Step to watch the window (lo…hi) shrink. Blue = lo, purple = hi, yellow = mid.

Ready.

Binary search O(log n)needs sorted input

Best O(1)Avg/Worst O(log n)Space O(1)
python
def binary_search(arr, target):
    lo, hi = 0, len(arr) - 1
    while lo <= hi:
        mid = (lo + hi) // 2        # avoids overflow in most languages: lo + (hi-lo)//2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            lo = mid + 1            # target is in the right half
        else:
            hi = mid - 1           # target is in the left half
    return -1
Use when the array is sorted, or you can binary-search on an answer (see patterns).
Avoid when data is unsorted (sorting first costs O(n log n)) or changes constantly.

⚠️ Classic bugs: wrong <= vs <, forgetting mid ± 1 (infinite loop), and integer overflow on (lo+hi) in languages with fixed ints.

04 The simple sorts — Bubble, Insertion, Selection

These are O(n²) and too slow for big data, but they're where everyone learns sorting — and insertion sort is genuinely useful for tiny or nearly-sorted arrays.

Pick an algorithm and press Sort. Yellow = comparing, red = swapping/placing, green = locked in place. Watch how the efficient sorts settle far faster.

comparisons 0 · swaps 0

Bubble sort O(n²)

Repeatedly swap adjacent out-of-order pairs; the largest "bubbles" to the end each pass. Mostly a teaching tool.

Best O(n)Worst O(n²)Stable ✔

Selection sort O(n²)

Each pass finds the minimum of the rest and puts it next. Few swaps (O(n)), but always O(n²) comparisons.

All cases O(n²)Swaps O(n)Stable ✗

Insertion sort O(n²) worst · O(n) nearly-sorted

Build a sorted prefix by inserting each new element into its right place, like sorting a hand of cards. Excellent for small or nearly-sorted data — real sort libraries switch to it for tiny sub-arrays.

Best O(n)Worst O(n²)Space O(1)Stable ✔
python
def insertion_sort(a):
    for i in range(1, len(a)):
        key, j = a[i], i - 1
        while j >= 0 and a[j] > key:   # shift bigger elements right
            a[j + 1] = a[j]
            j -= 1
        a[j + 1] = key                # drop key into the gap
    return a

05 Merge sort ★

Merge sort O(n log n) alwaysstable

Idea (divide & conquer): split the array in half, sort each half recursively, then merge the two sorted halves in linear time. Guaranteed O(n log n) in every case — the reliable choice, and the standard for sorting linked lists and huge/external data.

Best/Avg/Worst O(n log n)Space O(n)Stable ✔
python
def merge_sort(a):
    if len(a) <= 1:
        return a
    mid = len(a) // 2
    left  = merge_sort(a[:mid])       # sort left half
    right = merge_sort(a[mid:])       # sort right half
    return merge(left, right)

def merge(l, r):
    out, i, j = [], 0, 0
    while i < len(l) and j < len(r):   # weave two sorted lists
        if l[i] <= r[j]: out.append(l[i]); i += 1
        else:            out.append(r[j]); j += 1
    out.extend(l[i:]); out.extend(r[j:])
    return out
Use when you need guaranteed speed, stability, or you're sorting linked lists / data too big for memory.
Avoid when memory is tight — it needs O(n) extra space (quicksort sorts in place).

06 Quick sort ★

Quick sort O(n log n) avgO(n²) worst

Idea: pick a pivot, partition the array so smaller elements go left and larger go right, then recurse on each side. Sorts in place and is usually the fastest in practice — which is why it's the default in many standard libraries (with tweaks).

Best/Avg O(n log n)Worst O(n²)Space O(log n)Stable ✗
python
def quick_sort(a, lo=0, hi=None):
    if hi is None: hi = len(a) - 1
    if lo < hi:
        p = partition(a, lo, hi)      # pivot lands in final spot
        quick_sort(a, lo, p - 1)
        quick_sort(a, p + 1, hi)
    return a

def partition(a, lo, hi):
    pivot = a[hi]                     # Lomuto scheme
    i = lo - 1
    for j in range(lo, hi):
        if a[j] <= pivot:
            i += 1
            a[i], a[j] = a[j], a[i]
    a[i + 1], a[hi] = a[hi], a[i + 1]
    return i + 1
🛡️ The O(n²) worst case hits when the pivot is always the smallest/largest (e.g. already-sorted input with a naive pivot). Fixes: random pivot or median-of-three. Real libraries use introsort (quicksort that falls back to heap sort when recursion gets too deep).

07 Heap sort & the binary heap

Heap sort O(n log n) alwaysin place

A binary heap is a tree (stored in an array) where each parent is ≥ its children (max-heap). It gives you the max/min in O(1) and insert/remove in O(log n) — that's a priority queue. Heap sort builds a heap, then repeatedly pops the max to the end.

All cases O(n log n)Space O(1)Stable ✗

Why it matters: the heap / priority queue is the real prize — it powers Dijkstra's algorithm, Huffman coding, event simulations, "top-K" queries, and task schedulers. In interviews, reach for a heap whenever you hear "k largest / smallest / closest" or "merge k sorted…".

09 Breadth-First Search (BFS) ★

BFS O(V + E)uses a queue

Explore a graph level by level from a start node using a queue. On an unweighted graph, BFS finds the shortest path (fewest edges). Think: ripples spreading on water.

Time O(V + E)Space O(V)Shortest path (unweighted) ✔
python
from collections import deque

def bfs(graph, start):
    seen = {start}
    q = deque([start])
    order = []
    while q:
        node = q.popleft()            # FIFO — nearest first
        order.append(node)
        for nb in graph[node]:
            if nb not in seen:
                seen.add(nb)
                q.append(nb)
    return order
Use for shortest path in unweighted graphs, level-order traversal, "minimum steps" puzzles, flood fill.
Avoid for weighted shortest paths — that needs Dijkstra.

10 Depth-First Search (DFS) ★

DFS O(V + E)stack / recursion

Go as deep as possible along one path, then backtrack. Implemented with recursion (the call stack) or an explicit stack. The backbone of cycle detection, connected components, topological sort, and most backtracking.

Time O(V + E)Space O(V)
python
def dfs(graph, node, seen=None):
    if seen is None: seen = set()
    seen.add(node)
    for nb in graph[node]:
        if nb not in seen:
            dfs(graph, nb, seen)      # dive deeper before siblings
    return seen
Use for cycle detection, connected components, topological order, maze/grid exploration, path existence.
Avoid for shortest paths — DFS doesn't find them; and watch stack depth on huge graphs (use an iterative stack).

11 Dijkstra's shortest path ★

Dijkstra O((V + E) log V)needs a min-heap

Finds the shortest path from a source to every node in a graph with non-negative weights. It's a greedy algorithm powered by a priority queue: always expand the closest unfinished node next. This is the map-app / network-routing classic.

Time O((V+E) log V)Space O(V)Non-negative weights only
python
import heapq

def dijkstra(graph, src):             # graph[u] = [(v, weight), ...]
    dist = {u: float('inf') for u in graph}
    dist[src] = 0
    pq = [(0, src)]                   # (distance, node) min-heap
    while pq:
        d, u = heapq.heappop(pq)
        if d > dist[u]:               # stale entry — skip
            continue
        for v, w in graph[u]:
            if d + w < dist[v]:       # found a shorter route
                dist[v] = d + w
                heapq.heappush(pq, (dist[v], v))
    return dist
⚠️ Dijkstra breaks with negative edges. For those, use Bellman-Ford O(V·E) (also detects negative cycles). For all-pairs shortest paths on small graphs, use Floyd-Warshall O(V³).

12 Topological sort

Topological sort O(V + E)DAGs only

Order the nodes of a directed acyclic graph so every edge points "forward" — i.e. every task comes after its prerequisites. Course scheduling, build systems (Make/npm), and spreadsheet recalculation all rely on it.

Kahn's algorithm (BFS style): repeatedly take a node with in-degree 0, output it, and remove its outgoing edges. If nodes remain but none has in-degree 0, the graph has a cycle (no valid order).

python
from collections import deque

def topo_sort(graph):                 # graph[u] = list of successors
    indeg = {u: 0 for u in graph}
    for u in graph:
        for v in graph[u]:
            indeg[v] += 1
    q = deque([u for u in graph if indeg[u] == 0])
    order = []
    while q:
        u = q.popleft(); order.append(u)
        for v in graph[u]:
            indeg[v] -= 1
            if indeg[v] == 0:
                q.append(v)
    return order if len(order) == len(graph) else None  # None ⇒ cycle

13 Union-Find (Disjoint Set Union)

Union-Find ≈ O(1) per opamortised

Tracks a collection of items grouped into disjoint sets, answering "are a and b in the same group?" and "merge these two groups" almost instantly. With path compression + union by rank, each operation is nearly O(1) (inverse-Ackermann α(n)).

Where it shines: Kruskal's minimum spanning tree, detecting cycles while adding edges, network/friend-connectivity, and grid "islands" problems.

python
class DSU:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n
    def find(self, x):
        while self.parent[x] != x:
            self.parent[x] = self.parent[self.parent[x]]  # path compression
            x = self.parent[x]
        return x
    def union(self, a, b):
        ra, rb = self.find(a), self.find(b)
        if ra == rb: return False     # already connected (a cycle if adding an edge)
        if self.rank[ra] < self.rank[rb]: ra, rb = rb, ra
        self.parent[rb] = ra
        if self.rank[ra] == self.rank[rb]: self.rank[ra] += 1
        return True
🌉 Minimum Spanning Tree (cheapest set of edges connecting everything): Kruskal = sort edges, add each if it doesn't form a cycle (Union-Find). Prim = grow the tree greedily with a heap. Both are must-knows for "connect all cities at minimum cost".

14 Recursion & divide-and-conquer

Recursion = a function that calls itself on a smaller sub-problem until a base case stops it. Divide-and-conquer is the pattern: split the problem, solve the pieces recursively, combine the results (merge sort, quick sort, binary search all use it).

  • Always define the base case first — the smallest input you can answer directly. Miss it and you get infinite recursion / stack overflow.
  • Trust the recursion: assume the recursive call is correct for smaller inputs, then just combine.
  • Cost often follows the Master Theorem: splitting into 2 halves + linear combine ⇒ O(n log n).
python
def factorial(n):
    if n <= 1:            # base case
        return 1
    return n * factorial(n - 1)   # recursive case

15 Dynamic Programming (DP) ★

DP solves problems with overlapping sub-problems by solving each sub-problem once and reusing the answer — turning exponential brute force into polynomial time. The most feared interview topic, but it's one idea: remember what you've already computed.

Two flavours, same idea

  • Memoization (top-down): write the natural recursion, then cache each result in a dictionary/array. Compute on demand.
  • Tabulation (bottom-up): fill a table from the smallest sub-problems upward. Usually a bit faster, no recursion depth limit.
python
# Fibonacci: naive recursion is O(2^n); memoized DP is O(n).
from functools import lru_cache

@lru_cache(maxsize=None)              # top-down memoization
def fib(n):
    if n < 2: return n
    return fib(n - 1) + fib(n - 2)

def fib_tab(n):                       # bottom-up tabulation, O(1) space
    a, b = 0, 1
    for _ in range(n):
        a, b = b, a + b
    return a

0/1 Knapsack

Maximise value of items packed under a weight limit; each item taken or not. O(n·W)

python
def knapsack(wt, val, W):
    n = len(wt)
    dp = [[0]*(W+1) for _ in range(n+1)]
    for i in range(1, n+1):
        for w in range(W+1):
            dp[i][w] = dp[i-1][w]        # skip item i
            if wt[i-1] <= w:             # or take it
                dp[i][w] = max(dp[i][w],
                    val[i-1] + dp[i-1][w-wt[i-1]])
    return dp[n][W]

Longest Common Subsequence

Longest sequence appearing (in order) in two strings — basis of diff and DNA alignment. O(m·n)

python
def lcs(a, b):
    m, n = len(a), len(b)
    dp = [[0]*(n+1) for _ in range(m+1)]
    for i in range(1, m+1):
        for j in range(1, n+1):
            if a[i-1] == b[j-1]:
                dp[i][j] = dp[i-1][j-1] + 1
            else:
                dp[i][j] = max(dp[i-1][j], dp[i][j-1])
    return dp[m][n]
🔑 How to spot a DP problem: you're asked for a min / max / count / "is it possible", choices compound, and a brute-force recursion would re-solve the same sub-problems. Other classics to know: coin change, longest increasing subsequence, edit distance, matrix path sums, subset sum.

16 Greedy algorithms

Greedy usually O(n log n)

Build the answer by always taking the best-looking choice right now, never reconsidering. Fast and simple — but only correct when local choices lead to a global optimum (you must be able to prove it, or it can silently give wrong answers).

Must-know greedy wins: interval / activity scheduling (pick the earliest-finishing meeting), Huffman coding (optimal compression), Dijkstra, Kruskal & Prim (MST), and fractional knapsack.

python
# Activity selection: attend the most non-overlapping meetings.
def max_activities(intervals):        # intervals = [(start, end), ...]
    intervals.sort(key=lambda x: x[1])   # sort by earliest finish
    count, end = 0, float('-inf')
    for s, e in intervals:
        if s >= end:                  # doesn't clash — take it
            count += 1
            end = e
    return count
Use when you can prove the greedy choice is safe (exchange argument), and you need speed.
Avoid when choices interact — e.g. 0/1 knapsack. Greedy fails there; use DP.

17 Backtracking

Backtracking exponential, but pruned

Systematically build candidates one choice at a time; the moment a partial choice can't possibly work, undo it and try the next ("prune"). It's DFS over the space of solutions. The go-to for puzzles and "generate all valid…" problems.

Signature problems: N-Queens, Sudoku solver, generating all permutations / subsets / combinations, word search, and the "restore IP addresses" style questions.

python
# All subsets (the power set) via backtracking.
def subsets(nums):
    res, path = [], []
    def dfs(i):
        if i == len(nums):
            res.append(path[:])       # record a complete choice
            return
        dfs(i + 1)                    # choice: skip nums[i]
        path.append(nums[i])          # choice: take nums[i]
        dfs(i + 1)
        path.pop()                    # backtrack — undo the choice
    dfs(0)
    return res
✂️ The whole art is pruning: the earlier you detect a dead end, the faster it runs. Good pruning turns "impossible" brute force into "instant."

18 Hashing & hash tables

Hash table O(1) average lookup

A hash function maps a key to an array index, giving average O(1) insert, delete, and lookup. This is the dictionary / hash map / set you use every day — and the single biggest "cheat code" in interviews for turning O(n²) into O(n).

  • Collisions (two keys → same slot) are handled by chaining (linked lists) or open addressing (probing).
  • Worst case is O(n) if everything collides, but with a good hash it's effectively O(1).
  • Keys must be hashable / immutable (strings, numbers, tuples).
python
# Two-sum in O(n): remember what you've seen in a hash map.
def two_sum(nums, target):
    seen = {}                         # value -> index
    for i, x in enumerate(nums):
        if target - x in seen:        # complement already seen?
            return [seen[target - x], i]
        seen[x] = i
    return []
💡 Interview reflex: "find a pair / duplicate / count of…" ⇒ a hash set or hash map almost always cuts a nested loop down to a single pass.

19 String pattern matching (KMP & Rabin-Karp)

Finding a pattern in text O(n + m)

Naively searching for a pattern of length m in text of length n is O(n·m). Two classic algorithms fix that:

  • KMP (Knuth-Morris-Pratt) — precomputes a "failure/LPS table" so that after a mismatch it never re-checks characters it already matched. Guaranteed O(n + m).
  • Rabin-Karp — hashes the pattern and each sliding window of the text; compares hashes first (a "rolling hash" updates in O(1)). Great for multiple-pattern search and plagiarism detection. O(n + m) average.
📚 You rarely re-implement these from scratch in interviews, but you should recognise them by name and know they beat the naive O(n·m). The rolling hash idea also powers many "substring" problems.

20 Number-theory essentials

Euclid's GCD O(log n)

Greatest common divisor by repeated remainder — 2,300 years old and still optimal.

python
def gcd(a, b):
    while b:
        a, b = b, a % b
    return a

Fast exponentiation O(log n)

Compute a^n by squaring — essential for big powers and cryptography (with a mod).

python
def power(a, n):
    result = 1
    while n:
        if n & 1: result *= a   # bit set ⇒ multiply
        a *= a                  # square the base
        n >>= 1                  # shift to next bit
    return result

Sieve of Eratosthenes O(n log log n)

Find every prime up to n by crossing out multiples. The fastest simple way to list many primes.

python
def primes_up_to(n):
    is_prime = [True] * (n + 1)
    is_prime[0] = is_prime[1] = False
    for p in range(2, int(n**0.5) + 1):
        if is_prime[p]:
            for m in range(p*p, n + 1, p):   # cross out multiples of p
                is_prime[m] = False
    return [i for i, ok in enumerate(is_prime) if ok]

21 Interview patterns worth their weight

Beyond named algorithms, these reusable patterns crack a huge share of interview problems. Recognising the pattern is half the battle.

👉 Two pointers

Two indices moving through a (usually sorted) array — pair sums, removing duplicates, reversing, container-with-most-water. Turns O(n²) into O(n).

🪟 Sliding window

A moving sub-array/sub-string window that grows and shrinks — "longest substring without repeats", "max sum of k elements". O(n).

➕ Prefix sums

Precompute running totals so any range sum is O(1). Basis of many subarray-sum and 2-D grid questions.

🎯 Binary search on the answer

When the answer is monotonic ("can we do it in ≤ X?"), binary-search over the answer itself — "minimum capacity to ship in D days", "Koko eating bananas".

📚 Stack for structure

Matching brackets, "next greater element", evaluating expressions, monotonic-stack problems.

🔺 Heap for top-K

"k largest / closest / most frequent", "merge k sorted lists" — a size-k heap gives O(n log k).

22 Interview Q&A — quick-fire

Why is quicksort often faster than merge sort if both are O(n log n)?
Quicksort sorts in place with excellent cache locality and small constant factors, while merge sort needs O(n) extra memory and does more data movement. Quicksort's downside is the O(n²) worst case, which good pivot choice (random / median-of-three) makes vanishingly unlikely.
When would you use BFS over DFS?
Use BFS for the shortest path / fewest steps in an unweighted graph and for level-order processing. Use DFS for path existence, cycle detection, connected components, topological sort, and backtracking. Both are O(V+E); the difference is order of exploration.
What makes a problem solvable by dynamic programming?
Two properties: optimal substructure (the best answer is built from best answers to sub-problems) and overlapping sub-problems (the same sub-problems recur). If sub-problems are independent, plain divide-and-conquer suffices; if greedy choices are provably safe, greedy is simpler.
Why can't Dijkstra handle negative edge weights?
Dijkstra "finalises" a node the moment it's popped as the closest, assuming no cheaper route can appear later. A negative edge could create a cheaper route after finalisation, breaking that assumption. Use Bellman-Ford (O(V·E)) for negative edges — it also detects negative cycles.
Hash table lookup is O(1) — so what's the catch?
It's O(1) on average. Worst case is O(n) if many keys collide into one bucket (or a hostile input targets the hash). Good hash functions, load-factor resizing, and treeing long buckets keep it effectively constant in practice.
Greedy vs DP for the knapsack problem?
Fractional knapsack (you can take part of an item) is solved greedily by value-per-weight. 0/1 knapsack (whole items only) needs DP — greedy gives wrong answers because taking a high-ratio item can block a better combination.
How do you find a cycle in a graph?
Undirected: DFS and check for a visited neighbour that isn't the parent, or use Union-Find while adding edges (an edge joining two nodes already in the same set = cycle). Directed: DFS with three colours (white/grey/black) — a back-edge to a grey node means a cycle; or run Kahn's topo sort and see if any node is left unprocessed.
What does "stable sort" mean and when does it matter?
A stable sort keeps equal elements in their original relative order. It matters when sorting by one key after another (e.g. sort by name, then stably by age → names stay ordered within each age). Merge and insertion sort are stable; quick and heap sort are not.

23 A study roadmap that works

1

Complexity first

Get comfortable reading Big-O. You can't compare algorithms without it.

Binary search + merge/quick sort. Implement them from memory until it's automatic.

Data structures
